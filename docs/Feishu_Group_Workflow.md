# GeoGi 飞书群工作流

版本：v1.0

## 1. 两个机器人分工

### GeoGi客户小助手

类型：飞书群内创建的自定义机器人。

用途：

- 向飞书群发送通知
- 通知客户提交成功
- 通知品牌信息补齐和问题生成完成
- 通知报告初稿生成完成

限制：

- 只能发送消息
- 不能接收群里 @ 指令

### GeoGi诊断小助手

类型：飞书开放平台开发者后台创建的应用机器人。

用途：

- 后续接收群里 @ 指令
- 触发品牌信息补齐和问题生成
- 触发报告生成
- 查询项目状态
- 读写飞书多维表格

当前本地联调阶段，先用 `/api/feishu/command` 模拟群指令。备案和公网 HTTPS 完成后，再把飞书事件订阅接到 `/api/feishu/events`。

## 2. 标准业务流程

### 第一步：客户提交资料

客户在小程序提交资料后，系统自动完成：

1. 写入 `客户与品牌`
2. 创建 `诊断项目`
3. 向飞书群发送通知

状态：

- `客户与品牌`：`待人工审核`
- `诊断项目`：`待人工审核`

此时不自动生成信源、关键词、热门问题和AI检测问题。

### 第二步：人工审核

运营人员在飞书里检查：

- 品牌名称
- 企业主体
- 官网或官方渠道
- 行业和细分业务
- 核心产品或服务
- 主要客户
- 竞品或对标品牌

### 第三步：触发品牌信息补齐和问题生成

群指令：

```text
@GeoGi诊断小助手 开始品牌信息补齐和问题生成 项目编号 GG-P-202607-123456
```

本地模拟：

```bash
cd server
pnpm workflow:command "开始品牌信息补齐和问题生成 项目编号 GG-P-202607-123456"
```

系统自动完成：

1. 创建或跳过已有 `品牌基础档案`
2. 生成 `全网信源`
3. 生成 `品牌关键词`
4. 生成 `行业热门问题`
5. 生成 `AI检测问题`
6. 向飞书群通知生成结果

### 第四步：AI平台问答

运营人员在5个平台完成问答：

- DeepSeek
- Kimi
- 豆包
- 通义千问
- 腾讯元宝

优先从 `AI检测问题` 表里选择问题进行测试。

### 第五步：触发报告生成

推荐格式：

```text
@GeoGi诊断小助手 开始生成报告 项目编号 GG-P-202607-123456

DeepSeek：
链接：https://example.com
提问：我们公司想找一家做品牌AI可见度诊断的服务商，有没有比较靠谱的推荐？
回答原文：……

Kimi：
链接：https://example.com
提问：……
回答原文：……
```

如果只有会话链接，也可以先发送链接。系统会先尝试自动读取公开分享页的问答正文，再写入工作台：

- `已自动读取`：可作为本次诊断依据
- `部分读取`：只读到部分正文或摘要，会写入记录，但需要人工复核
- `读取受限`：平台要求登录、验证或没有暴露正文，需要补充截图、导出文本或更换公开链接

当前已验证：Kimi、豆包、腾讯元宝通常可以从公开分享页读取正文；DeepSeek 可能只能读取到部分内容；通义千问公开分享页可能只暴露标题和摘要。

系统自动完成：

1. 写入 `平台测试记录`
2. 写入 `回答分析结果`
3. 写入 `报告管理`
4. 更新项目阶段
5. 向飞书群通知报告状态

### 第六步：查询项目状态

群指令：

```text
@GeoGi诊断小助手 查询项目状态 项目编号 GG-P-202607-123456
```

系统返回：

- 客户状态
- 项目阶段
- 信源数量
- 关键词数量
- 行业问题数量
- AI检测问题数量
- 测试记录数量
- 分析结果数量
- 报告数量

## 3. 正式接入飞书事件回调

备案和服务器上线后：

1. 将服务器域名配置为 HTTPS
2. 在飞书开放平台开启 `GeoGi诊断小助手` 机器人能力
3. 将 `GeoGi诊断小助手` 添加到工作群
4. 在事件订阅里配置请求地址：

```text
https://api.geogi.cn/api/feishu/events
```

5. 订阅群消息或机器人被 @ 的消息事件
6. 使用同一套工作流处理器执行指令
